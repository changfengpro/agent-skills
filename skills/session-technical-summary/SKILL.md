# 会话技术总结

将当前 AI 对话，或本机最近一段时间内的 AI 会话记录，归档为可沉淀到技术文档中的 Markdown 与 HTML 记录。

## 触发规则

- 只有用户明确点名 `session-technical-summary`，或明确说“调用/使用这个 skill”时，才执行本技能。
- 不要因为一次对话结束就自动执行。
- 普通的“总结一下”不触发本技能，除非用户同时点名 `session-technical-summary`。
- 如果用户点名 `session-technical-summary` 并要求“今日/近几日/指定日期范围”的会话总结，执行本技能的“范围归档模式”。
- 如果用户没有点名本技能，只是要日报、工作日报或邮件日报，应使用专门的 `daily-summary` 工作流。

## 两种模式

### 当前会话归档

用于用户要求总结“本次对话/当前会话”时。直接根据当前上下文生成总览总结和本次时间线条目，然后调用写入脚本。

硬边界：

- 只能总结当前 `CODEX_THREAD_ID` / 当前聊天线程里的内容。
- 不要读取、合并或复用同日期下其他会话目录的 `summary.md` / `summary.html`。
- 只有当某个既有 `summary.md` 的 `Session ID` 与当前 `CODEX_THREAD_ID` 完全一致时，才能把它当作当前会话的既有归档读取。
- 如果发现同一日期目录下存在其他会话归档，只能忽略；不要把旧会话的问题详情并入当前会话总览。

### 范围归档

用于用户明确点名本技能，并要求总结“今日”“昨天”“近两日”“近几日”“最近 N 天”或具体日期范围内的本机会话时。

日期解释规则：

- `今日`：本地日期的 00:00 到当前时间。
- `昨天`：昨天 00:00 到 23:59:59。
- `近 N 日` / `最近 N 天`：包含今天，向前取 `N - 1` 个自然日。例如 2026-05-28 的“近两日”表示 2026-05-27 到 2026-05-28。
- 用户给出明确起止日期时，按用户日期为准。
- 如果用户说“过去 N 小时”，按滚动时间窗口理解，不按自然日。

范围归档应先从本机日志抽取证据，再撰写总结，不要只凭当前对话记忆。

## 输出目录

默认归档根目录使用当前系统用户的 home 目录推导：

```text
$HOME/文档/AI-Session
```

因此不同机器或不同用户名下会自动落到各自用户目录，例如：

```text
/home/alice/文档/AI-Session
/Users/alice/文档/AI-Session
```

当前会话归档的文件结构示例：

```text
$HOME/文档/AI-Session/YYYY-MM-DD/<session-folder>/summary.md
$HOME/文档/AI-Session/YYYY-MM-DD/<session-folder>/summary.html
```

这只是默认示例。用户指定其他根目录时，使用脚本的 `--root` 覆盖默认值。

范围归档也使用同一目录体系，默认写到“调用当天”的日期目录下，`<session-folder>` 使用范围标识，例如：

```text
$HOME/文档/AI-Session/YYYY-MM-DD/range-2026-05-27_to_2026-05-28/summary.md
$HOME/文档/AI-Session/YYYY-MM-DD/range-2026-05-27_to_2026-05-28/summary.html
```

`<session-folder>` 的选择规则：

- 当前会话归档时，优先使用第一次对话在 Codex/Claude 中显示的会话标题，并将其转换为文件名安全的目录名。
- `CODEX_THREAD_ID` 只写入 `Session ID` 元数据，不要把 UUID 作为默认目录名。
- 如果无法读取显示标题，再使用用户提供的标题；仍然没有标题时才使用临时时间戳兜底。
- 范围归档时使用 `range-YYYY-MM-DD_to_YYYY-MM-DD` 这类范围标识作为目录名。

## 文档结构

`summary.md` 顶部必须是整个会话截至当前的总览总结，下面按时间线追加每次调用本技能后的单次总结：

```markdown
# 会话标题

- 文档类型: AI 会话技术总结
- 日期: YYYY-MM-DD
- Session ID: `...`
- 首次创建: ...
- 最近更新: ...
- 输出目录: `...`

## 总览总结

这里是截至当前的全局技术复盘。每次调用后应重写这部分，使它反映整个会话的最新状态。

## 调用时间线

### 第 1 次调用 - YYYY-MM-DD HH:MM:SS +0800

这里仅记录第 1 次调用时新增归档的事实。

### 第 2 次调用 - YYYY-MM-DD HH:MM:SS +0800

这里仅记录第 2 次调用后新增或修正的事实，不重复第 1 次已经记录过的内容。
```

## 增量原则

执行本技能时，先读取已有 `summary.md`，再生成本次写入内容。

- `总览总结`：重写为截至当前的整体技术复盘，可以整合前文，但保持简洁，不堆砌每次调用的细节。
- `调用时间线`：只追加本次调用新增的事实、修正、证据、决策和后续事项。
- 不要在新的时间线条目中重复已经归档过的旧内容，除非是在修正旧结论。
- 如果需要修正旧内容，在本次条目中明确写“修正：...”，不要静默改写历史时间线。
- 命令输出只摘录关键行，但保留命令、路径、设备名、错误信息、退出码、文档链接等可追溯信息。

## 总览总结写作视角

总览总结不要写成“本会话围绕……”“我们讨论了……”“用户做了……”这类带对话主语的叙述。采用无主语的技术复盘写法，先给一段能直接上报日报的精炼总结，再按问题拆成详情。

顶部必须先写 `### 日报上报简述`：

- 3-5 条要点即可。
- 每条都要能独立用于日报上报。
- 逻辑顺序建议为：推进目标、关键问题、处理结果、未完成风险、下一步。
- 保持精炼，不堆命令输出；具体命令和证据放到问题详情。

随后写 `### 问题详情`，下面每个问题用 `#### 问题：...` 作为小标题。HTML 渲染器会把这些问题标题自动收集到侧边栏，点击后跳转到对应详情。

每个问题详情按固定小标题组织：

- 背景
- 问题表现
- 处理过程
- 结果
- 后续

推荐组织方式：

```markdown
### 日报上报简述

- ...
- ...

### 问题详情

#### 问题：ATK-DLRV1126B SDK 构建日志如何判断失败性质

##### 背景

- ...

##### 问题表现

- ...

##### 处理过程

- ...

##### 结果

- ...

##### 后续

- ...
```

禁止在总览总结正文中出现带操作者或聊天场景的主语和套话。执行前自检并删除这类表达。

避免：

- “本会话围绕...”这类会话视角开头。
- “用户在...”这类操作者视角开头。
- 只罗列主题，不说明问题和处理结果。
- 把工具验证输出孤立堆在最后，而不说明它证明了什么。
- 把未验证事项写成已完成。

## 内容质量要求

总结必须能作为技术文档素材继续整理。优先保留有证据的信息：

- 会话目标、任务边界、项目/设备背景。
- 关键操作、命令、工具调用和结果。
- 文件、文档、服务、定时器、串口、开发板设备或其他产物路径。
- 错误现象、根因判断、约束条件和技术结论。
- 当前状态、未解决问题和下一步建议。

使用简洁的中文技术表达。不要写没有证据支撑的“已完成”“已修复”“验证通过”。

## 执行流程

### 当前会话归档流程

1. 用本地时间确定日期 `YYYY-MM-DD`。
2. 确定 session ID；当前 Codex 会话默认由脚本从 `~/.codex/state_5.sqlite` 读取第一次对话显示的标题。
3. 只查找当前 session ID 对应的既有 `summary.md`。如果同日期目录下有其他会话归档但 `Session ID` 不一致，必须忽略。
4. 生成两个 Markdown 片段：
   - `overall.md`：截至当前的总览总结，必须使用第三方技术文档作者视角说明“做了什么、出了什么问题、如何处理、结果如何、后续是什么”。
   - `update.md`：本次调用新增的时间线条目正文。
5. 调用脚本写入并渲染：

   ```bash
   python3 ~/.agents/skills/session-technical-summary/scripts/write_session_summary.py \
     --date YYYY-MM-DD \
     --session-id "${CODEX_THREAD_ID:-manual-session}" \
     --overall-file /path/to/overall.md \
     --input /path/to/update.md
   ```

   也可以用管道传入 `update.md` 正文；`--overall-summary` 可直接传入较短的总览总结。只有在脚本无法读取会话显示标题时，才手动传入 `--title`。

6. 向用户报告实际写入的 `summary.md` 和 `summary.html` 路径。

### 范围归档流程

1. 按用户话术确定日期范围。
2. 用采集脚本抽取本机 Claude Code / Codex 会话证据：

   ```bash
   python3 ~/.agents/skills/session-technical-summary/scripts/collect_session_range.py \
     --start-date YYYY-MM-DD \
     --end-date YYYY-MM-DD \
     --json > /tmp/session-range.json
   ```

   也可以用 `--days N` 表示包含今天的近 N 个自然日。

3. 阅读 JSON，基于证据生成两个 Markdown 片段：
   - `overall.md`：日期范围内所有会话的总览总结，按项目/主题合并，并用技术复盘视角说明问题、处理方式、结果和后续。
   - `update.md`：本次范围归档调用新增的时间线条目，只描述这次采集到的日期范围、数据源、关键发现和后续事项。
4. 写入范围文档：

   ```bash
   python3 ~/.agents/skills/session-technical-summary/scripts/write_session_summary.py \
     --date 调用当天YYYY-MM-DD \
     --session-id "range-YYYY-MM-DD_to_YYYY-MM-DD" \
     --title "AI 会话技术总结（YYYY-MM-DD 至 YYYY-MM-DD）" \
     --overall-file /path/to/overall.md \
     --input /path/to/update.md
   ```

5. 范围总结必须按时间线组织关键会话，不要把原始日志直接堆进文档。保留能支撑结论的命令、路径、错误、文档链接和产物路径。

范围归档建议结构：

```markdown
#### 本次新增摘要

- 本次归档覆盖 YYYY-MM-DD 至 YYYY-MM-DD，共采集 Claude Code X 个会话文件、Codex Y 个线程 / Z 个 rollout。

#### 按时间线归档的会话要点

- YYYY-MM-DD HH:MM，项目/目录：...

#### 关键技术结论

- ...

#### 后续事项

- ...
```

## 单次时间线条目建议格式

```markdown
#### 本次新增摘要

- ...

#### 关键操作与证据

- ...

#### 技术结论

- ...

#### 后续事项

- ...
```
